# Проект: iOS-разработка с ролями в Cursor

**Оплата ИИ:** только подписка Cursor (без отдельных API-ключей провайдеров для «агентов»). Xcode и Apple Developer — отдельно, это стек Apple.

## Как это работает

В Cursor нет отдельных параллельных ботов. **«Агенты» — это роли:** набор правил в `.cursor/rules/*.mdc` и ваши формулировки в чате. Модель читает этот файл и правила, вы задаёте **задачу** и при необходимости **фазу** («сначала только архитектура, без правок UI»).

- **Повсюду:** правило потока работы `ios-workflow.mdc` подсказывает, когда идти по шагам, а когда можно один проход.
- **По типу файлов:** подключаются `swift-swiftui`, `ios-testing`, `ios-platform-privacy`; архитектурные ориентиры — в `ios-architecture.mdc` (включено для `.swift`).

Рекомендация: для крупной фичи — **разные чаты** (архитектура → реализация → тесты) или явные фазы в одном чате.

## Стек проекта (зафиксировано для агента)

**Минимальная iOS:** 18.0+ (сменить при необходимости).

**Интерфейс:** SwiftUI; UIKit только через `UIViewControllerRepresentable`, если без этого нельзя.

**Архитектура:** **MVVM**. Экран = тонкая `View` + **`Observable` ViewModel** (или `@Observable` класс) с явным `@MainActor`, если модель живёт в UI-слое. Бизнес-правила и разбор ответов API не клади в `View`; тяжёлую логику выноси в отдельные типы (сервисы / use case), которые ViewModel вызывает через **протоколы**.

**Навигация:** одно согласованное решение на проект — например координатор / роутер / единый стек `NavigationPath` на фичу. Не размазывай решение о переходах по десятку view без явной задачи на рефакторинг.

**Зависимости:** внедряй через протоколы + инициализатор (или готовый DI-контейнер, если уже есть); не обращайся к синглтонам из кода фич **без** явной договорённости в задаче.

**Слои (направление зависимостей):** View → ViewModel → абстракции сервисов → реализации (сеть, хранилище). ViewModel не импортирует конкретный `URLSession` — только протокол клиента.

**Модули:** SPM targets (или структура папок фич); имя корневого таргета приложения и список пакетов — по факту репозитория.

**Сеть:** `URLSession` + отдельный слой маппинга DTO → доменные модели; ошибки — типизированные, на границе UI — человекочитаемое сообщение по политике продукта.

**Хранение:** по умолчанию **SwiftData или UserDefaults** для простых настроек; чувствительное — **Keychain**; **Core Data** — только если уже в проекте или явно указано в задаче.

**Не смешивать** с TCA, VIPER или «чистым» Clean в новой фиче без отдельной постановки на миграцию.

## Структура команды (роли)

1. **Product / Discovery** — постановка, критерии приёмки, краевые случаи. Триггер: неясная задача. Выход: сценарии, без кода.
2. **Lead Architect** — слои, навигация, DI, контракты между фичами. Триггер: новая фича, новый модуль.
3. **Client (SwiftUI)** — экраны, состояние, concurrency, a11y, локализация. Триггер: UI и связка с доменом.
4. **Data & Networking** — API, DTO, ошибки, границы кэша. Триггер: новый endpoint, модели.
5. **QA** — XCTest, моки, регрессия. Триггер: после контракта; перед релизом; баг с воспроизведением.
6. **Platform & Compliance** — Keychain, ATS, фон, Privacy Manifest, подготовка к ревью. Триггер: данные пользователя; фон; релиз.
7. **Performance** — main thread, перерисовки, `os_log`. Триггер: лаги, большие списки.
8. **Build & Release** — схемы, конфиги, TestFlight. Триггер: первый билд, регресс сборки.

## Задача → как формулировать запрос

- Мелкий фикс: «одним проходом: …»
- Новая фича: «Фаза 1: только границы модулей и протоколы. Фаза 2: реализация. Фаза 3: тесты на границах.»
- Только ревью: «Ревью как Platform: проверь PrivacyInfo и ATS для …»
